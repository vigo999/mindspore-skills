# ACLNN 算子开发全流程（MindSpore 适配）

**凡使用本 skill 的 AI 必须遵守下列「核心行为准则」**

---

## 核心行为准则

> **第一层级为最高优先级**；本准则不依赖工作区是否配置 `.cursorrules`，执行时不得被其他软性建议带偏。  
> **心智顺序**：先「想对」（认知红线）→ 再「做对」（执行协议）→ 再「能交付」（交付标准）→ 最后「可进化」（进化策略）。  
> 本 skill 的流程、模板、命名约定可能随 MindSpore 版本迭代而过时，每一步须以仓库现状为准。

### 第一层级：认知红线

> **目的**：本层级是后续所有开发行为的逻辑底座，任何偏离都会导致工程规格失效。对抗幻觉，确保「判断力」正确。

1. **三位一体校验**
   - **核心逻辑**：严禁单一来源盲从。决策必须基于 **[仓库现状]、[接口定义]、[工程规范]** 三方对齐。
   - **版本前瞻**：MindSpore 迭代快，严禁依赖陈旧知识。每一步须 `grep` 确认当前分支真实写法（Infer API、注册宏等），优先对齐近 6 个月内合入代码。
   - **辩证参照**：参考同目录相似算子，但禁止盲从。若源码与接口定义（头文件）或项目规范冲突，以定义与最新规范为准，并向用户提示源码中的潜在遗留问题。
2. **确定性原则**
   - **无证不推**：接口/依赖须有物理证据（grep/头文件），**严禁臆造**。
   - **疑点上报**：当文档/头文件/源码冲突时，**强制停止**，列出冲突点请求仲裁。
3. **属性正交与层级隔离**（即：不同维度的属性不能互相推导，高优先级声明覆盖低优先级说明）
   - **语义解耦**：API 属性（类型/值域/必选性）严格独立，禁止跨维度推演。
   - **效力分层**：核心声明（签名/强约束）优于次级说明（备注/建议）；例如参数的硬性约定（必选性、类型、值域、默认值等）须以接口签名或规范中的显式声明为准，避免仅凭建议性表述（如「建议值」「推荐值」）推断。
4. **工程直觉**
   - 在执行任何修改时，若发现潜在的逻辑冗余、隐性耦合或语义模糊，应主动提出重构建议，确保系统的整体熵值不随功能增加而线性上升。

### 第二层级：执行协议

> **目的**：确保「交付力」，防止多文件联动导致的结构性遗漏。

5. **跨文件变更一致性（D1）**
   - **联动范围**：涉及 **8 类文件**（op_def YAML、api_def YAML、GeneralInfer C++、PyBoost C++、KBK kernel C++、bprop C++、Python 接口层、中英文文档）；同一参数名/类型/默认值/约束会在上述文件中多处引用。任何单点修改必须执行**动作闭环**：定位变更点 → 全量检索 → 同步更新 → 残留验证。
   - **四步动作**：
      1. **定位变更项**：明确本次修改的具体修改点（如参数名 `actual_seq_len`、dtype 约束 `float16/bfloat16`、dispatch 配置 `Ascend: XxxAscend`）。
      2. **全量检索**：在同一批操作中，对整个算子相关目录执行关键词搜索，定位所有引用点。
      3. **同步更新**：逐一修改所有引用点，确保命名形态（PascalCase vs snake_case）转换正确且逻辑一致。
      4. **残留验证**：修改完成后再次搜索旧值，确认全路径零残留。

6. **多维交叉验证（D3）**
   - **对标一致性**：**必须对标 PTA（PyTorch Adapter）**，含边界行为与报错信息，为规格达标硬指标。
   - **多视角校验**：
     - **实现视角**：代码逻辑是否正确、与相似算子实现是否一致。
     - **评审视角**：对照 `checklists.md` 对应章节逐项核对，每项须可判定「是/否」。
     - **用户/场景视角（Step 8 测试阶段必须）**：典型使用场景、业界论文或大模型中的调用方式与输入规格是否在用例中覆盖；当前接入要求为**对标 PTA 一致**，与 PTA 对标通过即视为满足。其余步骤可不单独列出，因评审视角的 PTA 对标项已覆盖与业界实现一致。
   - **验证记录**：实现视角与评审视角的验证结果均需在「验证闭环」模板中显式记录；Step 8 还须在验收/测试报告中记录用户/场景视角的结论（或明确说明已通过 PTA 对标满足）。

### 第三层级：交付标准

> **目的**：确保「合规性」，通过 CI 门禁。格式错误（如 clang-format 不通过）会直接阻断 PR，故为**从「做完」到「能交付」的硬性门槛**，非可后置的「打磨」。

7. **工程格式规范（D2）**
   - **硬性门槛**：产出须预先通过 Pylint/Clang-format/Markdownlint 等检查，为 PR 合入的物理前置。
   - **物理指标**：UTF-8、Tab 缩进、禁 Windows 路径、单行 ≤120；合规性自检含 Check_Scanoss（开源合规）。
   - **Skill 维护规范**：见 `reference.md` §31。

### 第四层级：进化策略

> **目的**：确保「成长性」，防止同类错误复发。

8. **问题定位与经验沉淀（D4）**
   - **触发条件**（满足任一即执行）：同类问题重复出现（≥2 次）；修复中发现连锁遗漏；问题根因在流程或规范层面而非代码层面。
   - **闭环动作**：
     1. **定位环节**：确认问题发生在哪个开发环节（YAML 定义 / Infer 推导 / kernel 实现 / 接口导出 / 测试覆盖 / 文档同步）。
     2. **逆向复盘**：确认修复所依赖的关键前置信息，评估该信息是否已在 skill 文档中显式记录。
     3. **经验规则化**：将教训转化为 `checklists.md` 中可独立判定「通过/不通过」的检查项，实现 Skill 自我迭代。

9. **维护 skill 高效可用性（熵增控制）**
   - **精简原则（去重）**：禁止添加含义重复或存在包含关系的规则。若新规则能覆盖旧规则，则删除/合并旧项。
   - **指令脱敏（去噪）**：删除一次性的、特定算子的上下文特例，只保留具备泛化意义的硬性规范。
   - **结构扁平（降重）**：避免嵌套过深的逻辑。若某个子步骤过于复杂，应将其转化为独立的「工具化模板」或「Checklist」，而非不断堆砌在核心流程中。
   - **逻辑自愈**：当发现 Skill 的流程、模板与仓库现状背离时，必须优先修正 Skill 指引（遵循变更一致性），确保指令的时效性与物理可执行性。

### 两条接入路径（Pre-B 必须决策，详见 `reference.md` §2.3）

| | **路径 1：自动生成** | **路径 2：Customize** |
| --- | --- | --- |
| **判断条件** | API 参数能原样透传给 ACLNN | 调用 ACLNN 前需要参数预处理 |
| **YAML dispatch** | `enable: True`，**不写** `Ascend` | `enable: True` + `Ascend: XxxAscend` |
| **PyBoost/KBK** | 编译自动生成（跳过 Step 4/5 手写） | 开发者手写 customize 文件 |
| **Step 4/5** | **验证**即可（确认自动生成产物正确） | **必须手写**实现 |
| **典型算子** | abs、mul、trunc、xlogy | dense_lightning_indexer、conv2d_ext |
| **gen_ops.py 作用** | 生成完整 PyBoost/KBK 调用+注册代码 | 生成包装代码，调用你的 Customize 类 |

### Workflow 执行清单

使用此 skill 开发 ACLNN 算子时，按上方「回复与执行规范」创建待办、逐项完成、结束时产出操作记录。

  - 输出：存量检查结果、接口分析结论、方案文档、调用链盘点表（组合场景）

  - 这是评审和转测交付的**必须产物**，必须在 Pre-B 完成后、Step 1 开始前生成初始化版
  - 后续每完成一个 Step，回填 Feature 文档对应章节

  - 输出：op_def + api_def + function_doc YAML 文件

  - 输入：YAML 文件、接入路径（Pre-B 确定的路径 1 或 2）
  - 输出：gen_ops.py 运行成功、自动生成文件（functional_overload.py、aclnn_kernel_register_auto.cc 等）

  - 输出：Infer 实现文件
- [ ] **[Step 4](workflows/04-pyboost.md)**：PyBoost（Pynative） → 回填 Feature §7
  - **路径 1**：跳过手写，验证自动生成产物正确即可
  - **路径 2**：手写 customize 实现文件（参数转换 + ACLNN 调用）
  - 输入：YAML、ACLNN 调用细节
  - 输出：customize 实现文件（路径 2）/ 验证通过（路径 1）
- [ ] **[Step 5](workflows/05-kbk.md)**：KBK（Graph） → 回填 Feature §7
  - **路径 1**：跳过手写，验证自动注册正确即可
  - **路径 2**：手写 kernel 文件（GetWorkSpaceInfo + Launch + 注册）
  - 输入：YAML、PyBoost 参考
  - 输出：kernel 实现文件 + 注册（路径 2）/ 验证通过（路径 1）
- [ ] **[Step 6](workflows/06-bprop.md)**：BPROP 注册 → 回填 Feature §11
  - 输入：derivatives.yaml 分析、反向 kernel
  - 输出：bprop builder 注册代码
- [ ] **[Step 7](workflows/07-export.md)**：导出与占位
  - 输入：算子实现
  - 输出：ops 包导出、接口文件、非 Ascend 占位
- [ ] **[Step 8](workflows/08-testing.md)**：测试 → 回填 Feature §12
  - 输入：全部实现、PTA 对标
  - 输出：C++ UT（必须新建） + Python ST（必须新建或确认已有覆盖） + Python UT（推荐）
  - **⚠️ 完成前必须逐项确认三类测试文件的产出状态，不允许只写一类就跳过**
- [ ] **[Step 9](workflows/09-docs.md)**：文档
  - 输入：接口实现
  - 输出：英文 function_doc（Step 1 已创建，此处完善）+ **中文 RST（公开 API 必须）** + 接口列表更新（按字母序添加到 mindspore.xxx.rst）
  - **⚠️ 英文 doc YAML 不等于文档步骤完成——中文 RST 是独立产物，最容易遗漏**
- [ ] **Feature 文档定稿** `🔒不可跳过`：补齐 §13（代码改动）、§14（验收报告）、更新 §3 任务状态
  - 即使 Step 9/10 被跳过或推迟，Feature 文档也必须在代码开发完成后补齐
- [ ] **[Step 10](workflows/10-delivery.md)**：转测交付
  - 输入：全部代码/测试/文档 + **完整 Feature 文档**
  - 输出：RFC issue + PR + 验收 checklist

## 开发前的"信息收集"清单（缺一不可）
在动手改代码前，先收集/确认（能从上下文推断就直接推断）：
- **算子名**：对外 API 名、内部 Primitive 名、Ascend kernel 调度名。
- **接入路径决策**：分析 MS API 参数是否可直通 ACLNN——**路径 1（自动生成）** vs **路径 2（Customize）**
  （详见 `reference.md` §2.3）。此决策直接影响 YAML 的 `dispatch.Ascend` 配置和 Step 4/5 的工作量。
- **ACLNN 对接对象**：具体接入哪个 `aclnnXxx`（含可能的变体/Grad）以及对应 ACLNN 文档/约束。
  **注意**：PTA 可能不是直连单个大算子，而是串联多个 ACLNN 小算子；需从 C++ 代码中提取完整调用链
  并盘点 MS 侧覆盖情况（详见 Pre-C 与 `reference.md` §28）。
- **PTA 对标与源码审查（必做）**：审查 op-plugin 中三类关键文件（functions.yaml / derivatives.yaml / OpApi.cpp），
  提取接口签名、反向接线、ACLNN 调用细节。文档与源码不一致时，整理差异清单交用户确认。详见 `reference.md` §25。
- **版本矩阵**：torch/torch_npu/CANN 版本（不同版本支持范围会变动，必须先记录再对齐）。
- **文档不足时的补证**：必要时生成"PTA 探测脚本"跑出真实支持范围，再回填约束到 YAML/Infer/测试/文档。  
  **硬性要求**：凡需用户跑脚本确认（如 PTA 参数支持范围、精度零偏差、ST 执行、性能/显存）时，必须 **(1) 提供可执行脚本**（若为探测类，基于 `scripts/probe_pta_sparse_flash_attention.py` 生成当前算子的 `probe_pta_{op_name}.py` 并写出到仓库）；**(2) 给出运行命令**；(3) **明确结果记录位置**（如 Feature 文档 §14 某表、§8 某节，或固定文件路径），让用户把结果写回该处，便于后续直接读取、不依赖对话粘贴。
- **输入/输出**：rank、layout（BSND/TND/PA_BSND 等）、dtype、可选输入/关键字参数、返回值个数。
- **是否需要反向**：梯度输入/输出定义、哪些输入需要梯度、是否有非张量参数。
- **动态支持**：动态 shape / 动态 rank / unknown value 的推导策略。
- **对标实现**：torch_npu / 参考 feature 文档 / 仓库中已接入的同类算子（见下方"相似算子查找策略"）。

### 相似算子查找策略（`reference.md` §2.4）
不要硬编码某几个算子名作为参照。正确做法：
1. **判断功能/算法类别**：目标算子属于哪个族？（Attention / Loss / Norm / Optimizer / 激活 / 逐元素算术 / Reduce / 矩阵运算 / 索引gather / 卷积池化 / 通信并行 等）
2. **确定技术特征**：在同族内用输入布局、ACLNN 对接方式、是否有反向、接口形态、参数特殊性进一步筛选。
3. **在仓库中搜索同类**：按族名 + 技术特征在 `op_def/yaml/`、`customize/`、`api_def/` 中 grep。
4. **选 2-3 个最匹配算子**，对照其各目录的代码风格和结构。
5. 相似算子只用于确认**目录结构、宏名、注册方式、测试写法**；功能逻辑以 PTA 源码 + ACLNN 文档为准。

## 卡住时的排障升级路径
当某个步骤反复失败或无法推进时，按以下优先级逐步升级：
1. **回退确认前提**：重新检查上一步的输出是否正确（YAML 是否合规、Infer 是否编译通过等）。
2. **对照相似算子**：按上方「相似算子查找策略」找最匹配的已有实现，逐行对比差异点。
3. **检查版本/环境**：核对「信息收集清单」中的版本矩阵，确认 CANN/torch_npu/MindSpore 版本是否匹配。
4. **缩小复现范围**：构造最小可复现用例，排除无关因素。
5. **向用户确认**：明确告知卡在哪、已尝试了什么、需要用户提供什么信息（日志/环境/权限/设备运行结果）。**不要默默跳过，必须暂停等用户回复后再继续。**
6. **建议提 issue**：如果判断是 CANN/框架 bug，建议用户按 RFC 流程提交问题记录。

> **禁止**：不要在卡住时反复尝试相同的修改方式，不要跳过失败步骤继续后续步骤。

### 当发现用户采用了与 skill 不同的做法时

触发场景：用户主动说明自己改了代码、或你发现代码状态与上一步产出不一致。

处理方式：
1. 简要确认差异点，询问原因——至少覆盖以下维度：
   - **前置条件**：是否有 skill 未考虑到的限制或依赖？
   - **历史原因 / 团队规范**：是组内惯例、流程变更、还是 skill 指引本身有误？
   - **泛化范围**：这个做法只适用于当前算子，还是所有同类场景都应该这样？
2. 根据 D4 的触发条件判断是否需要沉淀（同类问题 ≥2 次 / 连锁遗漏 / 根因在流程层面），
   并确定沉淀目标——经验可能不只属于本 skill：
   - 仅 ACLNN 算子相关 → 更新本 skill（workflow / checklist / reference）
   - 超出本 skill 范围（适用于其他算子开发、或属于编码规范/评审惯例等通用层面）
     → 告知用户，由用户决定写入哪个 skill 或项目级配置
3. 如需更新 skill，按 D1 确保内部一致性

> 不是每次绕过都要沉淀——判断标准见上方 D4 触发条件与熵增控制。

## 常见坑快速规避
- **YAML/代码生成**：缺 `py_method`、keys 结构不符会导致 `gen_ops.py` 报错。
- **InferInfo API**：按项目已有用法写（如 `GetScalarValueWithCheck` / `GetArrayValue` / `HasUnknownValue` / `IsNone`），不要臆造 API。
- **Infer 职责**：只做推导，不做运行时合法性校验（合法性让 ACLNN/运行时处理）。
- **前后向拆分**：前向/反向、PyBoost/KBK 通常都要分文件，注册也各自独立。
- **Windows 编码**：英文 YAML 文档避免夹杂中文（可能触发编码问题）；中文放 RST。
