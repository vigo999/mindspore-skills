# Workflow 4: PyBoost (Pynative, C++)

PyBoost is the aclnn operator invocation flow for PyNative mode.

## Goal

Implement the ACLNN operator call on the PyBoost path.
**The workload of this step differs greatly depending on the integration path:**
- **Path 1 (auto-generated)**: `gen_ops.py` already generated the full call code, so this step only needs **validation**
- **Path 2 (Customize)**: you must handwrite implementation files under the `customize` directory

## Inputs

- **Integration path**: auto / customize
- **YAML definition**: parameter list, `dispatch` configuration
- **PTA source analysis**: ACLNN call details and parameter preprocessing logic
- **(Composite scenarios)** ACLNN call chain: sub-operator list and dependency order

## Outputs

- **Path 1**: validated auto-generated PyBoost call code
- **Path 2**: handwritten PyBoost implementation files `op_name_ascend_customize.cc/.h`

---

## Steps

### Path 1 Branch: Validate The Auto-Generated Artifacts

> If Pre-B determined Path 1 (direct parameter passthrough), **you do not need to handwrite PyBoost files**.
> You only need to validate that the call code generated by `gen_ops.py` is correct:

1. **Confirm the generated code exists**: check that the generated PyBoost call artifacts are present in the auto-generated directory
2. **Confirm the ACLNN call arguments are correct**: verify that the generated `LAUNCH_ACLNN(aclnnXxx, ...)` uses the same argument order and types as the ACLNN documentation

> **If the generated code is wrong** (for example argument mismatch),
> it means the arguments cannot be passed through directly and you must reevaluate the path and switch to Path 2.

### Path 2 Branch: Handwrite Customize Files

#### Step 0: Verify The ACLNN Interface

> If the interface information was already confirmed during Pre-B/Pre-C, such as when the user supplied docs or the PTA source has already been analyzed, reuse those conclusions and skip this substep.

Before calling an ACLNN interface that has not yet been confirmed:
1. **Search with `grep`**: run `grep -r "aclnnXxx" .` to find existing in-project calls or header declarations.
2. **Confirm the signature**:
   - **If found**: extract the parameter list, especially the workspace-related argument positions.
   - **If not found** (new operator / headers not in repo / macro-generated signature): **do not guess**. Ask the user to provide the ACLNN doc or a header snippet.
3. **Record evidence**: show the found code references or the user-provided docs in your report.

#### Step 1: Single-Operator Direct Call Pattern

Standard three-part structure (`reference.md#pyboost-reference`):
1. output tensor allocation
2. argument conversion (`tuple -> vector`, `None` handling, and so on)
3. two-stage ACLNN invocation (`LAUNCH_ACLNN` or the project's equivalent macro)

### Step 2: Composite Operator Pattern (Composing Small C++ Operator APIs)

When the target operator is implemented as a composition of multiple smaller operators (`reference.md#composite-pyboost-pattern`):
1. include `#include "mindspore/ccsrc/include/pynative/utils/pyboost/functions/auto_generate/functions.h"`
2. call small-operator C++ APIs such as `add()`, `mul()`, `sum_ext()`, and so on to compose the logic directly, **without manually calling `LAUNCH_ACLNN`**
3. set `bprop_expander: False` in YAML so each small operator handles autodiff by itself
4. if the large operator already has its own explicit bprop, use `RequireGradGuard(false)` to prevent the small operators from triggering autodiff twice

### Step 3: View Operator Pattern (Zero-Copy, `reference.md#view-operator-development`)

When the operator is a pure shape/strides transform such as `transpose`, `reshape`, `expand_dims`, or `slice`:

1. **Do not use** `LAUNCH_ACLNN` or a PyBoost Customize implementation; the framework handles the View path
2. **Implement** the strides calculation function (`{OpName}ViewBasicTypeCalc`) and register it
3. **File locations**: `ops/view/{op_name}_view_strides_calc.cc` and `ops/include/view/{op_name}_view_strides_calc.h`
4. **YAML configuration**: add `view: True` to the original operator YAML
5. For strides calculation logic, refer to the corresponding operator in PyTorch `aten/src/ATen/native/TensorShape.cpp`

> View-specific YAML (`{op_name}_view_op.yaml`) usually delegates its strides calculation directly to the original operator's strides calculation function.

### Step 4: Input Argument Conversion (`reference.md#pyboost-argument-normalization`)

- `tuple` / `list` -> `std::vector<int64_t>`
- optional `None` inputs -> define the `None` semantics and handle them consistently in PyBoost / Infer / KBK
- scalar parameters -> extract using the project's scalar wrapper pattern

### Step 5: Compare Against Similar Operators (Repository Reality Wins)

You **must** refer to current similar operator files in the same directory to keep macro use and helper function usage consistent.
> Macro names, headers, and helpers may change across versions. Do not copy examples from `reference.md` blindly.
> Always use the latest real operator code under `customize/` as the final reference.

Code skeletons are available in `reference.md#pyboost-skeleton` for single operators, `reference.md#composite-pyboost-pattern` for C++ API composition, and `reference.md#view-strides-calculation`, but **the repository's current code is the authoritative reference**.

## Success Criteria

**Path 1**:
- [ ] Confirm the auto-generated PyBoost call code exists and uses the correct arguments

**Path 2**:
- [ ] PyBoost forward Customize implementation is complete
- [ ] PyBoost backward Customize implementation is complete (if needed)
- [ ] Argument conversion is correct (`tuple` / `None` / scalar)
- [ ] In composite scenarios, intermediate tensor allocation is correct and the call order matches PTA
- [ ] Style matches existing implementations in the same directory

**View operators**:
- [ ] The strides calculation function is implemented correctly (shape / strides / offset)
- [ ] The original operator YAML has `view: True`
